## 1. 診断の定義

表面的な性格ではなく、その人が無意識に反応してしまう本能を起点に、そこから生まれる欲望、人前で身につけた振る舞い、そのズレ、ストレス時の反応、そしてそれをどう魅力・成果・人間関係に変えるかを可視化する診断。

一文定義(正式採用):**「あなたが本当は何を求め、人前でどう隠し、どんなふうにすれ違いやすいかが分かる診断」**
(「どこですれ違っているか」等、領域特定を含意する表現は使用しない。領域特定は精密版の価値として温存する)

マーケティング上の立て付け:「本能・欲望・行動パターンをもとに、あなたの反応の傾向を整理する自己理解ツール」。科学的実証・脳科学的証明の主張はしない。脳の三層は理解を助けるメタファーとして扱う。

## 2. 測定対象と持ち帰り構造

階層モデル:本能 → 欲望 → 欲望の出し方 → 社会で身につけた振る舞い → ズレ → 防衛反応 → 使いこなし方

持ち帰り構造:
- メインラベル=メインタイプ+欲望の出し方(2語構成)
- サブ傾向は補足表示(§6の条件を満たす場合)
- ズレ・防衛反応・使いこなしはラベル化せず詳細な注釈として提示

| # | 測定対象 | 区分 | 内容 |
|---|---|---|---|
| A | タイプ(欲望の優先順位) | 直接測定→算出 | 4タイプの相対強度、最優位と2番手 |
| B | 欲望の出し方 | 直接測定→算出 | 外に出す/内に抱える/使い分け。タイプ固有の文脈で測る |
| C | 振る舞い | 直接測定 | 対人場面での実際の振る舞い(ズレのP側) |
| D | ズレ | 算出 | 大きさ・方向・一貫性の3指標 |
| E | 防衛反応 | 直接測定 | ストレス時の対処行動の型(タイプから独立した反応軸) |
| F | 使いこなし:自覚 | 直接測定 | 気づけている実感(自己認識) |
| G | 使いこなし:運用 | 直接測定 | 活かせている実感(自己認識) |
| H | 成果化・暴走/抑圧場面・再現性 | 推定・可能性 | 測定しない。F×G×タイプ等から記述 |

## 3. 測定・算出・推定・可能性の境界(正式仕様)

すべての結果文は以下4段階のいずれかに区分し、根拠レベルに応じた表現強度で執筆する。

**レベル1 直接測定:** 回答から対象軸のスコアを直接得たもの。表現は「回答上では」「この場面では」「傾向が強く出ている」。自己申告であるため「必ずそうする」「本当の姿である」とは断定しない。

**レベル2 組み合わせによる算出:** 複数の直接測定を定義済みルールで組み合わせたもの(メインタイプ、サブ傾向、確信度、ズレ3指標、使いこなし段階、各フラグ)。表現は「回答全体を組み合わせると」「比較的」「傾向がある」。算出ルールと使用質問IDを追跡可能に保存する。

**レベル3 根拠付き推定:** 測定・算出結果にタイプ別テンプレートを適用した解釈(周囲から見られやすい印象、誤解されやすい理由、強みとして活きやすい方向、つまずきやすい条件、相手への伝わり方)。表現は「見られやすい」「現れやすい」「可能性がある」。本人や他者の感情・過去の原因・現実の成果を事実として断定しない。

**レベル4 可能性としての提示:** 測定していない領域への展開(恋愛・仕事・友人・家族での具体的行動、第一防衛後の第二反応、過去の形成原因、将来の成果、心理的問題)。表現は「似た場面では」「こう出ることがある」「当てはまる場面がないか振り返ってください」。**領域別の断定は該当する精密版モジュール受診後にのみ許可する。**

**根拠メタデータ:** 各結果文に内部で紐付ける:evidenceLevel / sourceQuestionIds / sourceScores / confidence / scenarioScope / wordingStrength。確信度が低い場合は表現を弱めるだけでなく、根拠の弱い推定項目の非表示も許可する。

## 4. 質問数と分岐

### ユーザー向け表記
- 診断開始画面:「所要時間:約10分前後」「途中保存できます」
- 総問数・現在問数(「17/40」等)・「最大◯問」等はユーザーに一切表示しない
- 画面はSTEP表示(「STEP 3/5」)+プログレスバーのみ
- 追加質問発動時のマイクロコピー:「あなたの回答に合わせて、いくつか確認させてください」
- 選択肢順:共通タイプ・比較・防衛はセッションseedでランダム化し、中断再開後も同じ順序を再現する。リッカート1〜5は順序を固定し、ランダム化しない

### 内部仕様
- 全ルート共通の最短:39問
- ハード上限:48問/v1.0実運用上限:47問

**確定ルート(39〜47問):**
共通タイプ12 → 出し方(タイプ別)4 → 防衛反応7 → ズレ(タイプ別5ペア)10 → 使いこなし(タイプ別)6 = 39問

追加確認(2問単位、優先順位順):
| 優先 | 対象 | 発動条件 | 追加数 |
|---|---|---|---|
| 1 | タイプ比較 | T1−T2が0〜2点 | +2、割れたら+2(計4) |
| 2 | 出し方 | 中間帯判定 | +2 |
| 3 | 使いこなし | 正方向と逆転の矛盾 | +2 |
| 4 | ズレ | 方向不明瞭 | +2(1ペア) |

予算原則:追加は2問単位。次の単位で48問を超える場合は発動せず、不確かさはレポート表現の弱めで吸収する。全条件発動時の実到達最大は47問(例:39+比較4+出し方2+使いこなし2=47、ズレ確認は表現側で吸収)。

確認質問は2フェーズ制とする。タイプ比較のみ共通12問直後にインライン発動する。出し方・使いこなし・ズレの確認は基本ブロック回答後の確認フェーズで、出し方→使いこなし→ズレの固定優先順位に従い、残り予算内で発動する。動的な優先順位変更はMVPでは行わない。

**低確信度ルート(39〜45問):**
共通12 → [比較0または4] → 汎用出し方4 → 防衛反応7 → ズレ(汎用5ペア)10 → 使いこなし(上位2タイプ各3問)6
- 比較なし:12+4+7+10+6=39問
- 比較あり:12+4+4+7+10+6=43問
- 追加はズレ方向不明瞭時の汎用確認ペアGZ6（+2問）のみ許可(最大45問)。独立した「信頼性再確認」質問は使用しない
- 低確信度ルートでも測定量は減らさない(汎用出し方は4問。タイプ別より精度が落ちる前提を内部フラグで保持)

## 5. タイプ判定ルール(v1.0仮閾値)

配点:共通12問は各1点を選択タイプに加算し、`baseTypeScores`として保存する。合計12点。T1≧T2≧T3≧T4。比較質問の結果は`comparisonScores`として別保存し、共通12問の素点へ加算しない。最終判定は`baseTypeScores`と`comparisonScores`から`resolvedType`を決定する。サブ傾向表示と共通質問の識別力分析には`baseTypeScores`のみを使用する。

判定順序(固定):
1. 団子状態・平坦プロファイル判定
2. 明確差判定
3. 上位2タイプ比較質問
4. 比較後の確定または低確信度判定
5. 後続ブロックでタイプ適合性を確認(§7)
6. 最終確信度を確定(§6)

仮閾値:
- **団子・平坦:** T1−T3≦2、またはT1≦4 → 比較を経ず低確信度ルートへ直行
- **明確差:** T1−T2≧3 → 共通質問時点で仮確定
- **僅差:** T1−T2が0〜2 → 比較質問発動
- **比較質問:** まず2問。2問同一タイプ→仮確定/1対1→追加2問/4問中3問以上同一→仮確定/2対2→低確信度ルート
- **サブ傾向:** T2≧3点かつT2が単独2位の場合に表示候補。T2とT3が同点の場合は表示しない。低確信度・タイプ不適合フラグ・信頼性フラグがある場合は表示を弱めるか、メインタイプとの併記形式にする

上記はすべてv1.0仮閾値であり、リリース後に以下のログで校正する:再診断一致率/フィードバック評価/タイプ不適合フラグ率/低確信度ルート発生率/比較質問発動率/タイプ別の偏り。

## 6. 確信度(ブロック別)

確信度は診断全体に1つではなく、ブロック別に内部保存する:
**typeConfidence / expressionConfidence / gapConfidence / defenseConfidence / utilizationConfidence**(各high/medium/low)

typeConfidenceの仮ルール:
- **high:** 共通12問のみでT1−T2≧3、かつタイプ不適合フラグなし、かつ重大な信頼性フラグなし
- **medium:** 比較質問を経て3対1以上で確定/または共通質問では明確だが後続回答に軽微な矛盾がある
- **low:** 団子状態/比較2対2/タイプ不適合フラグあり/重大な信頼性フラグあり

(比較質問で確定した結果はmediumであり、highには含めない)

他ブロックの仮ルール:
- expressionConfidence:high=表出項目の方向が一致し中間帯でない/medium=確認質問でswitch平均4以上となり使い分け型が確定/low=確認後も使い分けが確定しない、または適合項目と矛盾。確認後も未確定の場合は基本スコアの近い打ち出す側／内に燃やす側へ寄せ、使い分け型にはしない
- gapConfidence:high=型判定が確定(ズレ小・抑圧・演出・反転のいずれか)/medium=確認ペアを経て確定/low=方向不明瞭のまま予算超過、またはブロック内に信頼性フラグ
- defenseConfidence:high=最頻カテゴリが3回以上/medium=2回/low=分散が大きく最頻が2回未満
- utilizationConfidence:high=逆転項目との矛盾なし/medium=軽微な矛盾を確認質問で解消/low=矛盾が残存

確信度→表現強度の変換:high=当該レベルの標準表現/medium=頻度・程度の限定語を追加/low=可能性表現へ格下げ、または項目非表示。

## 7. タイプ不適合フラグ(複数信号判定)

単一信号(本音フレーム平均のみ等)では判定しない。以下の複数信号の一致で立てる:

1. タイプ適合確認の独立項目(欲望そのものの存在を確認する項目。各タイプ別ブロックに最低1問。例:「周囲に見せるかどうかとは別に、自分の中では◯◯したいという感覚が強い」)が低値(仮:2以下)
2. 共通12問のT1−T2得点差が小さい(仮:2以下)
3. タイプの核を別表現で確認する第二の適合信号、またはタイプ別ブロック全体との重大な不整合
4. 結果フィードバックでタイプの核に違和感が選ばれた(事後信号として蓄積)

本音フレームの低値と使いこなし項目の低値は、抑圧・自覚不足・直近1か月に該当場面がなかった可能性があるため、フラグ成立条件には使わず分析用の参考ログとして保存する。

フラグ成立の仮ルール:信号1が該当し、かつ信号2または3が該当した場合。信号4は事後検証に使用し、診断時の表示は遡及変更しない。

フラグ時の処理:診断は最後まで継続/メインタイプを強く断定しない/サブ傾向との併記形式に切り替え/タイプ固有の推定文を弱める/フィードバックと紐付けて保存/共通質問改修の分析対象にする。MVPでは途中の再ルーティングは行わない。

適合確認項目は「欲望を外に出しているか」ではなく「欲望そのものが本人の中に存在するか」を確認する質問とする(出し方との混同禁止)。

## 8. 信頼性フラグ(6種・個別保存)

| フラグ | 定義(仮閾値) | 扱い |
|---|---|---|
| 高速回答 | 1問あたり中央値回答時間1.5秒未満 | 主信号 |
| 位置連打 | 同一表示位置の連続選択(仮:8問) | 補助信号(選択肢順ランダム化のため位置≠意味) |
| 意味的単調 | 同一タイプ/カテゴリの選択肢のみを機械的に選択 | 主信号 |
| 5段階同値連続 | リッカート項目で同値連続(仮:8問) | 主信号 |
| 逆転項目矛盾 | 正方向と逆転項目の同時高値 | 主信号(使いこなし確認質問の発動条件) |
| 類似質問不一致 | 確認質問と元回答の不一致 | 主信号 |

単一フラグだけで結果全体を弱めない。複数フラグの重複、または該当ブロック内の重大な矛盾がある場合に、そのブロックの表現を弱める。全フラグは個別にログ保存する。

## 9. ズレ3指標

前提:全ペアをスコアリングキーで極性統一し、本音H・対人Pとも「欲望を出している度」(1〜5)に揃える。

```
d[i] = P[i] - H[i]        # 負=本音より抑えている / 正=本音より出している

M = median(abs(d[i]))     # 大きさ(0〜4)
neg = count(d[i] <= -1)
pos = count(d[i] >= +1)
active = neg + pos
C = (active == 0) ? null : max(neg, pos) / active   # 一貫性

判定(v1.0仮閾値):
  M <= 1 かつ active <= 1        → ズレ小
  neg >= 2 かつ pos >= 2         → 反転(場面によって切り替わる。「ズレがない」とは書かない)
  C >= 0.75 かつ neg > pos       → 抑圧
  C >= 0.75 かつ pos > neg       → 演出
  それ以外                       → 方向不明瞭 → 予算内なら確認1ペア(+2問)、
                                   予算超過時は可能性表現に弱める
```

強度区分:抑圧・演出と判定された場合はMにより軽度(M=1)/中度(M=2)/強度(M≧3)を内部付与する。ユーザー向けに数値は出さず、表現強度と課金フック選択に使用する。軽度を重大な問題のように扱わない。

外れ値:Mに中央値を用いて単一ペアの影響を抑制。|d|=4のペアが1つだけある場合は回答引用の候補にする(誤答か、最も鋭い個別性かのいずれかであるため)。

## 10. 防衛反応

- 全タイプ共通7問:3シナリオ(裏切り/人前での失敗/長期停滞)×2問+存在意義の否定1問
- 各シナリオの1問目=第一反応、2問目=直後の行動または次の対処
- **選択肢は4タイプの欲望の言い換えにしない。** 対処行動(反撃/距離/迎合/分析/自己責任化/証明/感情遮断/固まる)として設計し、バランスよく配置する
- 防衛反応はタイプから独立した反応軸として集計する。「タイプAだから防衛Aになる」設計は禁止
- 出力は単純最頻ではなく同率数を含めて判定する。最大3回以上かつ単独首位=第一防衛high、2回かつ単独首位=第一防衛medium、2回で2カテゴリ同率=併記low、それ以外は単一の第一防衛を出さず「複数の守り方を場面で使い分ける傾向」と表示する
- 2回が3カテゴリ以上並立した場合は、第一防衛を確定しない。次点は第一防衛が単独成立し、次点が2回以上の場合のみ表示する
- 第一防衛（一定の安定性を示す傾向）と、各シナリオで今回選ばれた反応（直接測定）は別データとして保持する
- 次点が同率の場合は同率カテゴリを配列で保持し、任意の1カテゴリへ確定しない。結果文では併記可能とする
- 各カテゴリについて選択回数・出現機会数・selectionRateを保持する。selectionRateは同率カテゴリ内の表示順にのみ使用し、単独第一防衛への昇格には使用しない
- 出現機会数3以下のカテゴリへ`opportunityLimited`を付与する。confidenceはhigh=3回答以上の単独首位、medium=2回答の単独首位、low=同率・1回答のみ・証拠不足を維持する
- 防衛設問の選択肢は、特定タイプの語彙(例:勝つ、負ける、意味、仲間、理解)へ寄せすぎない。カテゴリ付与は回答の目的ではなく、観察可能な対処行動が一意になるよう設計する
- 「本人と話す」など複数の目的に読める行動を、迎合・反撃など一つのカテゴリへ安易に割り当てない。迎合は自分の不満や希望を抑えて相手へ合わせる行動として定義する

存在意義の否定(1問)の根拠区分:
- 直接測定:提示された場面で選択した反応傾向まで
- 根拠付き推定:他の防衛質問・タイプとの組み合わせによる解釈
- 可能性:反応の継続、直後の行動、領域別の現れ方、第二防衛
- 「安定した反応の型」「存在意義防衛タイプ」等の確定表現は禁止

## 11. 使いこなし

- タイプ別6問(正方向4+逆転2)。自覚(気づけている実感)と運用(活かせている実感)を分けて測る
- 現実の成果・成熟度・客観性は測らない。自己認識のみを測る
- ユーザー表示名:「気づけている実感」「活かせている実感」(「〜度」は使用しない)
- 期間指定「実際のこの1か月」を維持
- 逆転項目矛盾時は確認+2問(エピソード想起型)。低スコアは欠点ではなく伸びしろとして記述

低確信度ルート:上位2タイプから各3問（各タイプのA1・R1・O1）を出題し、指標レベルで6問を合算スコアリング。レポート記述も指標レベルの語りとし、段階表現は粗い3段階にする。

4. 欲望の出し方
5. 周囲から見られやすい印象(レベル3表現)
6. ズレの通知(攻略ポイントの言い回し。不安喚起表現は禁止)
7. 防衛反応のチラ見せ1文
8. 有料レポート導線+相性共有導線

低確信度時は単一タイプを断定せず、併記形式で提示する。

### 課金フック
- 既定:ズレ通知を主フックとする
- **フォールバック順:ズレ小(M≦1かつactive≦1)の場合はズレフックが機能しないため、防衛反応→使いこなし、の順で代替フックを選ぶ**
- hookTypeと購入有無を紐付けてログ保存(将来の動的フック選択への切り替えに備える)
- 言い回しは攻略ポイントのフレーム固定

### フィードバック
- 位置:無料結果表示の直後、課金導線の前
- 必須1問:「この結果はどれくらい当てはまっていましたか?」(4段階)
- 任意:特に当たっていた項目/違和感があった項目(タップ選択、スキップ可)

## 13. 一撃の執筆規則

- 一撃は根拠付き推定(レベル3)として扱う。描写する行動の核は直接測定したタイプの核の言い換えに限定する
- 具体的な場面は例示であることを「たとえば」で明示する
- 鋭さは断定語ではなく場面の具体性で作る
- 文末表現はtypeConfidenceに応じて切り替える:

| 確信度 | 型 | 例 |
|---|---|---|
| high | 具体描写+問いかけ | 「たとえば会議で自分の案が通らなかったとき。表では頷きながら、頭の中ではもう次の勝ち筋を探している。そんな自分に覚えはありませんか。」 |
| medium | 傾向表現 | 「たとえば会議で自分の案が通らなかったとき、表では受け入れながら、頭の中では次の勝ち筋を探し始めることがありそうです。」 |
| low | 複数傾向の併記 | 「あなたの中には、結果を取り返そうとする反応と、一度距離を取ろうとする反応の両方が見えています。場面によって、前に出る自分と引く自分が入れ替わるのかもしれません。」 |

## 14. 有料レポート

構成:
1. 有料版の一撃(無料より一段深く、回答参照を織り込む)
2. 本能と欲望の詳細
3. 欲望の出し方
4. 周囲から見られやすい印象
5. 誤解されやすい理由
6. ズレの詳細
7. 防衛反応の詳細
8. 気づけている実感
9. 活かせている実感
10. 強みへ変換しやすい方向
11. つまずきやすい条件
12. 次に試す具体的な行動
13. 精密版モジュール導線(各章末配置。文言は「この章を、あなたの◯◯についてさらに測る(8問)」に固定。「続きを見る」系は禁止)

根拠レベルの割り当て:
- **直接測定・算出として語ってよい:** メインタイプとサブ傾向/欲望の出し方/本音と振る舞いのズレ/防衛シナリオで選択した反応/自覚・運用に関する自己認識/各ブロックの確信度
- **根拠付き推定として語る:** 周囲から見られやすい印象/誤解されやすい理由/強みとして活きやすい方向/つまずきやすい条件/相手にどう伝わりやすいか
- **可能性としてのみ語る:** 恋愛・仕事・家族・友人での具体的行動/第一防衛後の第二反応/過去の形成原因/現実の成果/将来の成功/心理的問題

領域別表現の統一:
- 通常版:「似た場面ではこう出ることがある」「仕事や恋愛でも同じ構造が出ることがあります」「当てはまる場面がないか振り返ってみてください」
- 精密版受診後:該当領域での回答に基づき、領域別傾向として記述可能

回答参照:最低3か所。形式は「あなたは◯◯の場面で、△△に近い反応を選んでいます」(「本人の言葉」とは書かない)。「次に試す行動」は助言であり根拠レベルの断定対象外だが、行動提案の前提となるタイプ記述はconfidenceに連動して弱める。

## 15. 相性機能

相性は**根拠付き推定**として扱う(二人の実際の関係性を測定したものではない)。双方のタイプと出し方にタイプ組み合わせ別テンプレートを適用し、見込みを推定する。

- 使用可能な表現:合いやすい/すれ違いやすい/伝わりにくくなりやすい/翻訳が必要になりやすい/補いやすい
- 使用禁止:実際に仲が良い/恋愛相性が高い/結婚に向いている/長続きする/相手はこう感じている

共有:URL・QRを主導線、結果コード手入力は補助。コードに含めるのはメインタイプ・出し方・サブ傾向・簡易相性計算に必要な最低限のみ。ズレ・防衛・使いこなし・本音系回答・有料レポート内容は含めない。

無料の簡易相性:合いやすいところ(1)/すれ違いやすいところ(1)/二人の関係を表す一言(点数は出さない。関係の性質を言い当てるラベル)/相手にも診断してもらう導線。

有料の相性深掘り:伝わりやすい接し方/ぶつかりやすい場面/自分側が気をつけたいところ/相手に知っておいてもらいたいこと(相手の浅い層のみに基づくためタイプレベルに留める)/関係を良くするヒント。データの実体は「自分側の深い層×相手の浅い層」。形式は「あなたが◯◯すると、このタイプの相手には△△として伝わる可能性があります」。相手の感情・本音・防衛反応は断定しない。両者購入・同意時のみ解放する双方向深掘りは次段階。

用語規則:見出し・本文は日常語。「噛み合いやすい点」「相互作用」「関係性最適化」「補完可能性」「コミュニケーション特性」等の抽象語は禁止。

## 16. 経時変化・精密版・ログ・バージョン・中断再開

### 経時変化
| 区分 | 対象 | 再診断時の扱い |
|---|---|---|
| 変わらない前提 | タイプ、本能の優先順位 | 別タイプが出たら「揺らぎ」として通知し確認質問を提案。タイプ変更とは表示しない |
| 変わり得る | 出し方、振る舞い、ズレ、防衛の出方、使いこなし | 差分を変化としてポジティブに表示。成長レポートの中核 |

### 精密版(任意・モジュール制)
- 有料レポートの必須条件にしない。通常診断のみで無料・有料・相性まで成立させる
- 恋愛/仕事/人間関係(各8問)+防衛深掘り(6〜8問)。仕事モジュールは法人版の土台に転用
- 通常版との差は文章量ではなく主張の解像度(領域別に測り直したデータが根拠として実在する)
- 案内は有料レポート各章末と再診断時のみ。購入前には案内しない

### ログ保存(初期実装必須)
各質問への回答(questionId, questionVersion, 回答値, 回答時刻)/各軸の内部スコアと確信度(ブロック別5種)/仮判定の推移・通ったルート・発動した追加確認/信頼性フラグ6種(個別)/タイプ不適合フラグと構成信号/最終結果/scoringVersion・resultTemplateVersion・diagnosisEngineVersion/フィードバック回答/hookTypeと課金有無/共有・相性・精密版利用/中断位置と再開有無

### バージョン管理
全設問にquestionIdとversion。スコアリング・テンプレート・エンジンにも独立version。結果レコードに使用全versionを記録し、分析は同一version内で行う。設問差し替えはA/B並行配信を基本とする。

### 中断・再開
MVP:localStorage保存+再開ボタン+進捗復元(保存期限14日)。次段階:メール/LINE再開リンク。

## 17. 法人展開時の注意点(併記)

個人向け実装時点で構造として壊さないこと。

データ4区分(法人版で各測定対象に設定):①本人に表示するか ②管理者に表示するか ③管理者向け出力(関係性エンジン)の内部計算に使うか ④内部利用する場合の丸め方

既定方針:
- 本人向け:個人版同等の深さまで表示可。本人は「管理者に表示される自分の項目一覧」をいつでも確認できる
- 管理者向け:タイプ・出し方・仕事上の行動傾向のみ。ズレ・防衛・使いこなしの内面スコアは直接表示しない
- 内部利用:ズレ・防衛・使いこなしは関係性エンジンの計算に使用可。出力は内面の記述を含まない行動助言のみに丸める
- 採用・査定・配置転換への単独使用禁止を規約に明記。本人同意フローとロールベースアクセス制御を初期要件とする
- データ保持期間を定め、本人からの削除請求に対応する手順を利用規約に明記する
- 法人版設問は文脈差し替えではなく仕事場面用の別バンク(精密版の仕事モジュールを土台に拡張)

誤認防止:個人・法人・相性の全レポート末尾に定型文を表示する。
「本診断は医療・心理臨床上の診断ではなく、自己理解とコミュニケーションのための参考情報です」

## 20. 4タイプ横断検証による追加確定事項（改訂3）

### 20-1. 形成理由の扱い

通常診断では、欲望や行動が形成された過去原因を直接測定しない。したがって、「拒絶を避けるため」「分からない自分を見せたくないため」「理想を壊されたくないため」等の形成理由は事実として断定しない。

- 使用可:「背景には◯◯がある可能性があります」
- 使用可:「◯◯を避ける形で出ることがあります」
- 使用禁止:「◯◯だから隠しています」

精密版で該当条件を追加測定した場合のみ、測定された領域の傾向として記述できる。

### 20-2. 防衛カテゴリAPの表示名

内部コード`AP`は維持する。ユーザー表示は「迎合」ではなく、**自分を引いて合わせる**とする。

関係修復、対話、謝罪そのものをAPへ分類しない。自分の不満・希望・境界を抑え、相手側へ合わせる行動のみをAPとして扱う。

### 20-3. タイプ別の介入目的

質問選択肢、結果テンプレート、相性テンプレートには、行動の見た目だけでなく、その行動が何を変えようとしているかを示す`interventionGoal`を内部保持する。

| タイプ | interventionGoal | 意味 |
|---|---|---|
| 勝ち筋 | change_the_position | 立場・形勢・結果を変える |
| つながり | restore_the_connection | 関係の温度・相互反応を戻す |
| 読み解き | make_it_understandable | 情報・構造を理解可能にする |
| 軸 | make_it_meaningful | 理想・意味・価値との一致へ近づける |

同じ「質問する」「異論を言う」という行動でも、介入目的が異なる場合は異なるタイプ信号として扱う。

### 20-4. ズレ強度の2軸化

抑圧・演出の内部強度は、中央値による大きさ`M`だけでなく、ズレが生じた場面数`breadth=active`も保持する。

- M=1かつbreadth=1〜2:一部の場面で少し
- M=1かつbreadth=3〜5:幅広い場面で少しずつ
- M=2かつbreadth=2〜3:いくつかの場面で明確に
- M=2以上かつbreadth=4〜5:幅広い場面で明確に
- M=3以上:強度表現は使用できるが、問題・病理として扱わない

ユーザーへ数値は表示せず、レポート表現とフック選択に使用する。

### 20-5. 有料結果の個別化アンカー

有料レポートの章10「強みへ変換しやすい方向」、章11「つまずきやすい条件」、章12「次に試す具体的な行動」には、少なくとも1つの`personalizationAnchor`を必須とする。

利用可能なアンカー:
- `maxGapPair`
- `primaryDefense`
- `awarenessUtilizationGap`
- `expressionPattern`
- `directAnswerReference`

同一アンカーのみで全章を生成せず、一つのアンカーの利用は最大2章までを原則とする。タイプ一般論だけで章10〜12を生成してはならない。

### 20-6. 横断検証結果

勝ち筋・つながり・読み解き・軸の各タイプについて、打ち出す型と内に燃やす型の回答パターンを作成し、無料・有料結果への接続を検証した。全4タイプで縦接続は成立した。

詳細は別紙`4タイプ横断結果検証_v1.0.md`を参照する。

## 20. 12ラベル無料結果テンプレート確定事項（改訂4追記）

別紙 `12ラベル無料結果テンプレート_v1.0.md` を、4タイプ×3出し方の無料結果基準として採用する。

### 出し方3区分

- 打ち出す型:欲望を言葉・態度・行動として外へ出しやすい
- 内に燃やす型:欲望自体は強いが、外へ出す行動は抑えやすい
- 使い分け型:相手・場面・安全度・勝算・重要度等の条件に応じて外への出し方を切り替える

使い分け型を「中間」「曖昧」「どちらでもない」とは表現しない。条件による切替傾向として独立した結果価値を持たせる。

### タイプごとの介入目的

- 勝ち筋:立場・形勢・結果を変える
- つながり:関係の温度・相互反応を変える
- 読み解き:情報・構造・理解可能性を変える
- 軸:意味・理想・価値との一致を変える

同じ行動を記述する場合も、介入目的に基づいてタイプを区別する。

### 無料結果の個別化

固定テンプレートのみで表示せず、ユーザー固有の回答から最大2つの個別アンカーを差し込む。

候補:
- maxGapPair
- gapPattern
- primaryDefense
- awarenessUtilizationGap
- directAnswerReference

無料版では有料版の材料を使い切らない。最大差ペアの詳細説明、防衛との複合解釈、使いこなしの段差の理由、具体的行動提案は原則として有料版に残す。

### 課金フックの追加ルール

1. ズレ中度以上:ズレを主フック
2. ズレ軽度かつbreadthが4以上:幅広い場面で少しずつ起きる差をフック
3. ズレ小:防衛反応をフック
4. 防衛確信度low:気づけている実感と活かせている実感の段差をフック
5. 課題軸が弱い場合:強みを再現できる条件をフック

フックは不安喚起ではなく、「まだ言語化されていない攻略ポイント」として提示する。

## 21. 12ラベル有料結果テンプレート確定事項（改訂5追記）

別紙 `12ラベル有料結果テンプレート_v1.0.md` を、4タイプ×3出し方の有料結果生成基準として採用する。

### 21-1. 固定長文ではなく合成型

有料結果は12本の固定レポートをそのまま表示する方式にしない。以下を合成して生成する。

1. タイプ固有層
2. 出し方固有層
3. ズレ層
4. 防衛・使いこなし層
5. 個別回答アンカー層

これにより、同じラベルでもズレ・防衛・使いこなしが異なるユーザーへ同一文面を返さない。

### 21-2. 有料版で必須となる追加データ

- `maxGapPair`
- `gapPattern`
- `gapMagnitude`
- `gapBreadth`
- `primaryDefense` または `defenseProfile`
- `awarenessUtilizationGap`
- `directAnswerReference` 3か所以上
- 個別データに基づく具体行動1つ

これらを使用しない場合、有料版として成立しない。

### 21-3. 章10〜12の個別化

「強みへ変換しやすい方向」「つまずきやすい条件」「次に試す具体的な行動」には、最低1つの個別化アンカーを必須とする。

同一アンカーのみで3章を生成しない。一つのアンカーは最大2章までを原則とする。

### 21-4. 低確信度有料版

低確信度ユーザーへ単一タイプの13章を生成しない。以下の専用8章構成へ切り替える。

1. 二つの欲望の一撃

### 21-5. 無料版との境界

無料版はタイプ・出し方・ズレ通知・防衛チラ見せまでとする。有料版では、最大差場面、ズレの強度と広がり、防衛の組み合わせ、自覚と運用の段差、具体的行動まで接続する。

有料版は無料版の文章を長くしたものではなく、使用データと主張の深さが異なるレポートとする。
